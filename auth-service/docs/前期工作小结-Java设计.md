# 前期工作小结 —— Java 设计

## 一、前期完成的主要工作及质量

### 1. 总体架构与分层

- **技术选型**：采用 Spring Boot 2.3.12 作为主框架，MyBatis-Plus 作为持久层，MySQL 8 作为关系库，Thymeleaf 作为服务端页面渲染，整体符合企业级 Web 应用的常见选型，便于与区块链、AI 风控模块集成。
- **分层结构**：已形成清晰的 Controller（API + 页面路由）、Service（业务逻辑）、Mapper（数据访问）、Entity（领域模型）、config（配置与全局异常）的分层；身份、审计、链上验证、风控等职责边界明确，便于维护与扩展。
- **模块划分**：根项目为 Maven 多模块结构，当前主业务集中在 `auth-service` 模块（digital-identity-auth），包结构按功能划分（如 `blockchain`、`risk`），符合单一职责与高内聚要求。

### 2. 身份与业务核心

- **DID 身份方案（方案 A）**：在应用内生成去中心化标识符，格式为 `did:blockdia:emp:{工号}`，工号唯一约束保证 DID 唯一；身份实体（Identity）包含工号、姓名、部门、职位、角色、DID、创建时间及链上存证字段，与数据库表 `did_identity` 一一对应，设计完整。
- **身份服务**：注册、按 ID/DID/工号/链上交易哈希查询、列表、修改角色、注销等接口齐全；注册与权限变更均与审计、链上存证联动，事务边界清晰（`@Transactional`），数据一致性有保障。
- **审计日志**：独立 `audit_log` 表与 `AuditLogService`，记录权限修改、DID 注销等操作，并与链上存证键 `audit_{id}` 对应，为后续操作认证与溯源提供基础，质量可靠。

### 3. 认证与权限控制

- **双轨登录**：管理员登录（`/api/login`）与测试用户登录（`/api/login/test`）分离，避免混用；Session 中区分 `admin` 与 `user`，管理员页（如权限管理、审计日志）在 Controller 层做登录校验，未登录重定向至登录页，逻辑清晰。
- **权限与角色**：角色限定为「员工」「超级管理员」，与业务文档一致；修改角色、注销 DID 均写审计并上链，满足可追溯要求。

### 4. 接口与前端支撑

- **REST API**：身份注册与查询、链上验证（身份/操作认证）、审计列表、风险检查等接口完整；返回结构统一（如 `success`、`message`、`data`/`risk`），便于前端与联调。
- **页面路由**：首页、登录（管理员/测试用户）、注册、查询、凭证、权限管理、审计、存证记录、链上验证、风险报告等页面齐全，由 Thymeleaf 模板 + 少量前端脚本完成，与后端 API 对应良好。
- **全局异常处理**：通过 `@RestControllerAdvice` 将未捕获异常转为统一 JSON，避免 500 导致前端仅显示「网络或服务异常」，提升了可观测性与排错效率。

### 5. 质量评价

- 代码结构清晰，命名规范，关键业务有注释；事务与异常处理到位，与区块链、风控模块的集成点明确，**整体完成度与可维护性较好**，满足毕设阶段对 Java 后端设计的要求。

## 二、存在的问题

1. **配置与安全**：管理员与测试用户密码、数据库连接等写于 `application.properties`，未区分环境（如 dev/prod），敏感信息易泄露；生产环境应使用配置中心或环境变量，并避免明文密码。
2. **接口鉴权**：除管理员页面做 Session 校验外，部分 API（如身份列表、修改角色、注销）未显式做权限校验，依赖前端或调用方自律，存在越权风险。
3. **JWT 未实际使用**：pom 中引入 JWT 依赖，但当前认证完全基于 Session，无 Token 签发与校验；若需前后端分离或跨域调用，现有设计需扩展。
4. **可测试性**：Service 层对区块链、风控等依赖多为直接注入，缺少接口抽象或 Mock 友好设计，单元测试覆盖不足，回归成本较高。
5. **日志与监控**：除区块链模块部分日志外，业务层日志不够统一，缺少请求 ID、操作人、耗时等关键信息，不利于线上问题排查与审计增强。

---

## 三、拟解决的方法

1. **配置与安全**：引入多环境配置（如 `application-dev.properties` / `application-prod.properties`），敏感项改为环境变量或外部配置；考虑使用 Jasypt 等对配置中的密码做加密存储，并在文档中说明部署要求。
2. **接口鉴权**：对「仅管理员」或「仅超级管理员」可操作的接口（如身份列表、修改角色、注销、审计列表）增加统一鉴权（如拦截器或 AOP），根据 Session 中角色拒绝未授权请求并返回 403。
3. **JWT 与 Session 并存**：若后续需要前后端分离或开放 API，可保留 Session 用于管理端，对移动端或第三方提供 JWT 签发与校验；先明确需求再决定是否在毕设范围内实现。
4. **可测试性**：对依赖链上、风控的 Service 抽取接口（部分已有，如 `ChainEvidenceService`），在测试中注入 Mock 实现；为核心业务（如身份注册、权限修改、链上验证逻辑）编写单元测试与少量集成测试，保证主流程稳定。
5. **日志与可观测性**：约定业务日志格式（如操作类型、用户、关键 ID、结果），在关键入口与链上/风控调用处补充日志；可选增加请求 ID（MDC）与耗时统计，便于后期与监控系统对接。
