# 前期工作小结 —— 区块链设计

## 一、前期完成的主要工作及质量

### 1. 链选型与集成方式

- **链平台**：采用 FISCO BCOS 2.x，通过官方 Web3SDK（web3sdk 2.4.1）与节点建立 Channel 连接，使用国密（GenCredential）生成链上身份，与国内企业链常见部署方式一致，便于答辩与演示环境搭建。
- **配置驱动**：通过 `application.properties` 中的 `fisco.enabled`、节点地址、群组 ID、证书路径、存证合约地址等集中配置，`BlockchainConfig` 仅在 `fisco.enabled=true` 时装配 Web3j、Credentials、GasProvider 等 Bean，**未启用时不连链、不依赖节点**，满足“仅写库不连链”的降级需求，设计合理。

### 2. 存证合约与业务映射

- **合约设计**：`EvidenceStorage.sol` 提供 `set(string key, string hash)` 与 `get(string key)`，key 为业务键、value 为内容哈希，结构简单、职责单一，与“身份/审计存证”需求一一对应。
- **键值规范**：身份注册存证 key 为 DID（如 `did:blockdia:emp:10001`），审计存证 key 为 `audit_{日志id}`，与 Java 侧 `ChainEvidenceServiceImpl`、`EvidenceStorageContract` 的调用约定一致，便于链上查询与验证接口复用。
- **合约封装**：`EvidenceStorageContract` 使用 Web3j 的 Function 编码调用 set/get，不依赖 BIN 部署；Gas 采用 StaticGasProvider 固定值，满足存证场景，实现简洁、可维护性好。

### 3. 存证内容与一致性

- **哈希规则**：身份内容哈希由 `Identity.buildContentString` 拼接 did、工号、姓名、部门、职位、角色、创建时间（统一到秒的格式），再 UTF-8 SHA256 取 hex；审计内容由 `AuditLog.buildContentString` 拼接操作类型、DID、姓名、详情、操作时间后同样做 SHA256。**注册/审计写入与链上验证读取采用同一套规则**，避免时间精度或字段顺序导致校验失败，质量可靠。
- **写入时机**：身份注册在入库后取 DB 实际记录再算 hash 上链；审计在插入日志后同样以 DB 记录为准算 hash 上链，保证库与链一致，设计严谨。

### 4. 链上验证能力

- **身份验证**：`/api/chain/verify/identity?did=xxx` 根据 DID 查库得当前凭证，计算 expectedHash，再调用合约 get(did) 得 actualHash，比较后返回 PASS / MISMATCH / NO_RECORD，逻辑清晰，可直接支撑“凭证与链上存证是否一致”的核验场景。
- **操作认证**：`/api/chain/verify/audit?txHash=xxx` 先按交易哈希查审计表，若无再查身份表，对命中记录算 hash 并与链上 get(key) 比对，支持身份注册与权限/注销两类存证的验证，满足操作可追溯、可验证的设计目标。
- **区块高度**：提供 `/api/chain/block-number` 供首页展示链上区块高度，增强“已上链”的可视化与可信感。

### 5. 降级与可维护性

- **未启用链**：当 `fisco.enabled=false` 或未配置合约地址时，`ChainEvidenceServiceImpl` 不装配，由 `ChainEvidenceServiceNoOp` 占位，所有 `saveEvidence` 返回 null，业务侧已兼容（上链结果为 null 时仅不写 txHash/blockNumber），**不影响本地开发与无链演示**，降级策略明确。
- **异常处理**：链上调用异常在 Service 与 Controller 层被捕获并转为友好提示（如“链上服务未配置或未就绪”“链上查询失败”），避免节点异常直接导致 500，用户体验与稳定性较好。

### 6. 质量评价

- 区块链模块与业务解耦良好，通过接口与条件装配支持“开关式”上链；存证与验证流程完整，哈希与键值规范统一，**前期区块链设计完成度高、可答辩、可扩展**，符合毕设要求。

## 二、存在的问题

1. **合约部署与版本**：合约地址需手工部署后写入配置，项目内未提供一键部署脚本或部署文档；合约升级或重新部署后需人工更新 `fisco.evidence-contract-address`，易遗漏。
2. **证书与路径**：证书依赖 classpath 下 `conf/ca.crt`、`sdk.crt`、`sdk.key`，不同环境（如本机与虚拟机节点）路径或文件名可能不一致，当前未做环境区分配置，新人或换机时易报连链失败。
3. **Gas 与性能**：Gas 使用固定值，高并发或链负载高时可能存在交易排队或失败，未做重试与动态 Gas 策略；大量存证场景下的批量上链与性能未做优化。
4. **链上数据与库的一致性**：若链上曾写入某 key 后业务侧未正确落库（如异常中断），会出现“链有记录、库无记录”或反之，当前未设计对账或修复流程，仅能通过人工核对。
5. **隐私与合规**：存证内容为哈希，明文不落链，符合常见隐私要求；但 key 使用 DID 或 audit_id，链上可枚举 key 与 hash 对应关系，若需更高隐私可考虑后续引入加密或零知识方案（可作为毕设展望）。

## 三、拟解决的方法

1. **合约部署与版本**：在 `docs` 或 `scripts` 中增加“存证合约部署步骤”文档，记录 FISCO Console 部署命令、获取合约地址及填入 `application.properties` 的步骤；可选提供简单脚本（如读取 abi/bin 并调用部署接口），便于答辩与换环境时复现。
2. **证书与路径**：支持通过环境变量或配置文件覆盖证书路径（如 `fisco.ca-cert` 等），并在文档中说明本机与虚拟机节点的典型配置；若多环境部署，可与 Spring Profile 结合（如 `application-fisco-dev.properties`）。
3. **Gas 与性能**：短期内保持静态 Gas，在文档中说明当前为演示/毕设场景；若时间允许，可对 `saveEvidence` 增加失败重试（如 1～2 次）与简单退避；批量上链可留作后续优化点，在论文或总结中体现为“可改进方向”。
4. **链库一致性**：设计简单的“存证对账”思路：定期或按需扫描库内带 chain_tx_hash 的记录，调用链上 get 比对 hash，不一致时打日志或告警；具体实现可作为扩展功能或论文中的运维设计章节。
5. **隐私与合规**：当前方案保持“只存 hash、key 为业务标识”的设计，在文档中明确说明“链上仅存哈希、明文不出域”；若答辩或论文需要，可在“未来工作”中简述可引入的加密存证或零知识验证思路，不强制在本次实现。
