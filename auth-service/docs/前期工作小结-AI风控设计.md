# 前期工作小结 —— AI 风控设计

## 一、前期完成的主要工作及质量

### 1. 风控整体架构

- **双轨评估**：采用“规则引擎 + 可选 AI 模型”的架构。规则引擎基于近期行为统计与加权评分，阈值 60 判中风险、80 判高风险，逻辑可解释、不依赖外部服务即可运行；AI 模型为 Python 提供的异常检测服务（Isolation Forest），接收 4 维特征并返回风险等级，**启用时优先采用模型结果，超时或失败则自动降级为规则**，保证可用性与可答辩性。
- **行为驱动**：风险评估完全基于“身份查询”“链上验证”两类行为的近期频次，与业务场景（防滥用、防爬、防撞库）一致，特征明确、易于在论文中描述与实验对比。

### 2. 行为记录与特征

- **行为记录**：`BehaviorRecordService` 按用户维度在内存中维护近期行为队列，记录类型为 `ACTION_IDENTITY_QUERY`、`ACTION_CHAIN_VERIFY`，并带时间戳；单用户保留条数及时间窗口（如 5 分钟）有上限，避免无限增长，线程安全（ConcurrentHashMap + 队列同步），实现质量可靠。
- **特征口径**：1 分钟内链上验证次数、1 分钟内身份查询次数、5 分钟内链上验证次数、5 分钟内身份查询次数，与 Python 端 `FEATURE_NAMES` 及规则侧权重计算一致，**规则与模型共用同一套特征**，便于对比“规则 vs 模型”的效果。

### 3. 规则引擎

- **评分公式**：对 1 分钟/5 分钟内的验证与查询次数赋予不同权重（如 1 分钟内验证权重 2.0、查询 1.0，5 分钟内相应降低），加权求和得到 score；阈值 60 触发中风险、80 触发高风险，并返回统一结构 `RiskResult`（是否风险、等级、分数、说明、来源 rule/model），设计简洁、可调参。
- **集成位置**：在身份查询 API、链上验证 API 的响应中附带当前用户的 `risk` 对象；风险检查独立接口 `/api/risk/check` 供风险报告页使用，前后端对接完整。

### 4. AI 模型与 Python 服务

- **模型选型**：Python 侧使用 sklearn 的 Isolation Forest（或类似异常检测算法）在模拟数据上训练，输出异常分数并映射为高/中/低，与规则阈值（0.6/0.8）对应，满足“异常检测 + 等级输出”的需求。
- **接口约定**：Java 通过 `ai.risk.model.url`（如 `http://localhost:5000/predict`）POST JSON 体（4 个特征），读取 `risk_level`；超时由 `ai.risk.model.timeoutMs` 控制，超时或异常时静默降级为规则，**不阻塞主流程**，鲁棒性好。
- **配置开关**：`ai.risk.model.enabled` 为 false 时仅用规则，不请求 Python；为 true 时先请求模型，再视结果决定是否采用，满足“无 Python 环境也能跑通系统”的部署需求。

### 5. 风险升级与拦截

- **升级逻辑**：`RiskEscalationService` 在中风险预警后累计“预警后操作次数”，达到设定次数（如 3 次）则升级为高风险并**拒绝**后续身份查询与链上验证请求；拒绝状态按用户 ID 持久在内存 Set 中，不随 Session 失效而恢复，防止恶意用户通过反复登录绕过。
- **清除策略**：仅在新登录/登出时清除当前 Session 的风险预警与计数，不清除已升级为“拒绝”的用户集合，逻辑清晰，与“先预警、再升级、再拦截”的产品预期一致。

### 6. 仪表盘与统计

- **DashboardStatsService**：按自然日统计“今日链上验证次数”“今日中高风险事件数”，在身份查询与链上验证入口及风险升级/拒绝处埋点，首页展示与风控效果可视化得到支撑，数据口径明确。

### 7. 质量评价

- 风控模块从行为采集、特征计算、规则评分、模型调用、风险升级到统计展示形成闭环；规则可解释、模型可开关、降级有保障，**前期 AI 风控设计完成度与可用性较好**，符合毕设阶段对“智能风控”的设计与实现要求。

## 二、存在的问题

1. **行为数据持久化**：当前行为记录仅存内存，服务重启后历史行为清空，无法做长期分析与模型重训；也无法按日/周统计行为分布，限制后续“基于真实日志优化模型”的扩展。
2. **规则权重与阈值**：权重和阈值（60/80）为经验值，缺少基于历史数据或 A/B 的调参过程，论文中若要做“规则 vs 模型”对比实验，需补充调参说明或仿真数据。
3. **模型训练数据**：Python 侧使用模拟数据训练，与真实业务分布可能存在差异，模型在真实场景下的泛化能力与误报/漏报率尚未系统评估。
4. **风控与业务耦合**：风控埋点分散在各 Controller 中，若新增接口或变更流程易遗漏；缺少统一的风控切面（如 AOP）或明确的风控清单，维护成本随接口增多而上升。
5. **拒绝后的恢复**：用户一旦被标记为“拒绝”，当前仅能通过重启服务清空内存 Set 恢复，无管理员解封或按时间自动降级机制，演示或误伤场景下不够灵活。

---

## 三、拟解决的方法

1. **行为数据持久化**：可选将行为记录异步写入数据库或日志表（如 user_id、action_type、timestamp），保留最近 N 天；既支持服务重启后不丢近期数据，又为后续“用真实行为训练/评估模型”提供数据基础，实现时注意写入异步与性能影响。
2. **规则权重与阈值**：在文档或论文中明确当前权重与阈值的取值依据（如参考同类系统或初版经验值）；若有时间，可用历史或仿真数据做简单网格搜索或敏感性分析，并在“实验与结果”中给出规则在不同阈值下的准确率/召回率对比。
3. **模型训练数据**：在 `python-risk` 的 README 或 docs 中说明当前为“模拟数据训练、用于流程验证”；若毕设要求体现模型效果，可增加“用小规模脱敏真实行为日志做二次训练或评估”的步骤，并在论文中说明数据来源与局限性。
4. **风控与业务解耦**：梳理所有需做风控的 API（身份查询、链上验证等），在文档中维护“风控接口清单”；若后续接口增多，可考虑用 AOP 或 Filter 对指定路径统一做行为记录与风险检查，减少重复代码与遗漏风险。
5. **拒绝后的恢复**：增加“管理员解封”能力，如后台提供按用户 ID 从 `blockedUserIds` 中移除的接口（需鉴权）；或设计“拒绝状态 TTL”，超过一定时间自动从 Set 中移除（需权衡安全与体验），并在说明文档中写明策略与适用场景。
